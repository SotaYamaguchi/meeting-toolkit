# 機能追加フロー

このリポジトリに新しい機能（主にCobraサブコマンド）を追加するための段階的なワークフロー。
各フェーズでユーザー承認を得てから次に進むこと。

機能リクエスト: $ARGUMENTS

## Core Principles

- **既存パターンを踏襲する**: 新しいやり方を発明しない。既存の類似コマンドと同じ構造で実装する
- **CLAUDE.md に従う**: プロジェクトの規約（テストパターン、日本語出力、linter設定など）を遵守する
- **TodoWrite で進捗管理**: 全フェーズを通じてタスクを追跡する
- **ユーザー承認なしに実装に入らない**: 設計フェーズで必ず承認を得る

---

## Phase 1: 既存パターンの確認

**Goal**: 類似コマンドのコードを読み、踏襲すべきパターンを把握する

**Actions**:
1. TodoWrite で全フェーズのタスクリストを作成する
2. `cmd/` 配下の既存サブコマンド定義を読み、以下を把握する:
   - コマンド構造（`Use`, `Short`, `Args`, `ValidArgsFunction`, `RunE`）
   - フラグの定義パターン（`--project`, `--config` など）
   - 補完関数の登録パターン
   - `init()` でのサブコマンド登録方法
3. `pkg/` 配下の対応するビジネスロジックの構造を把握する
4. 既存テスト（`main_test.go`, `cmd/*_test.go`）のパターンを確認する

---

## Phase 2: 明確化質問

**Goal**: 設計前に、判断が分かれるポイントをユーザーに確認する

**Actions**:
1. Phase 1 の調査結果をもとに、以下の観点で不明点を洗い出す:
   - コマンド名・サブコマンド階層の配置
   - 必要なフラグやオプション
   - エラーハンドリングの方針
   - 既存ロジックの切り出し・リファクタリングの要否
   - エッジケースの扱い
2. **質問を整理してユーザーに提示する**
3. **回答を待ってから次のフェーズに進む**

自明なことは聞かない。判断が分かれるポイントだけを簡潔に質問する。

---

## Phase 3: 設計提示

**Goal**: 実装計画を提示し、ユーザーの承認を得る

**Actions**:
1. 以下を含む設計をまとめる:
   - コマンド体系（`mtg <sub> <type> [flags]` の形式）
   - 動作フロー（ステップごとの処理内容）
   - 変更ファイル一覧と各ファイルの変更内容
   - `pkg/` に追加・変更する関数のシグネチャ
2. **設計をユーザーに提示し、承認を待つ**
3. **承認なしに実装に入らない**

---

## Phase 4: 実装

**Goal**: 承認された設計に基づいてコードを実装する

**DO NOT START WITHOUT USER APPROVAL**

**Actions**:
1. `pkg/` のビジネスロジックを先に実装する（既存関数のリファクタリングが必要な場合はここで行う）
2. `cmd/` のサブコマンド定義を追加する（既存の `newXxxCmd` 関数パターンに合わせる）
3. `init()` にサブコマンドを登録する
4. `cmd/root.go` のヘルプテキスト（Long description と例）を更新する
5. TodoWrite でタスクを逐次更新する

---

## Phase 5: テスト追加

**Goal**: 追加した機能のテストを書く

**Actions**:
1. `pkg/` の新しい関数のユニットテストを `main_test.go` に追加する（既存テストのパターンに合わせる）
2. `cmd/` パッケージ内の非公開関数は `cmd/*_test.go` にテストを追加する
3. テストでは一時ディレクトリ・ファイルを使い、`defer func() { _ = os.RemoveAll(tmpDir) }()` パターンでクリーンアップする
4. 正常系・エラー系の両方をテーブル駆動テストでカバーする

---

## Phase 6: ビルド・テスト・リント実行

**Goal**: すべてのチェックをパスすることを確認する

**Actions**:
1. `make build` でビルドが通ることを確認する
2. `make test` で全テストがパスすることを確認する
3. `make lint` でリンターの警告がないことを確認する
4. いずれかが失敗した場合は修正して再実行する

---

## Phase 7: セルフレビュー

**Goal**: 実装が設計どおりか、不要な変更がないかを確認する

**Actions**:
1. `git diff` で変更内容を確認する
2. 以下の観点でチェックする:
   - CLAUDE.md の規約に準拠しているか（日本語メッセージ、errcheckパターンなど）
   - 既存コードのスタイルと一貫しているか
   - 設計で合意した範囲外の変更が含まれていないか
   - 不要な import やデッドコードが残っていないか
3. 問題があれば修正し、Phase 6 を再実行する

---

## Phase 8: コミット & プッシュ

**Goal**: 変更をコミットしてリモートにプッシュする

**Actions**:
1. `git add` で変更ファイルをステージングする
2. コミットメッセージは既存のコミット履歴のスタイルに合わせる（`feat:`, `refactor:` など）
3. `git push` でリモートにプッシュする

---

## Phase 9: サマリー出力

**Goal**: 完了報告をユーザーに提示する

**Actions**:
1. 全 TodoWrite タスクを完了にする
2. 以下をまとめて出力する:
   - 追加した機能の概要
   - 変更したファイル一覧と変更内容の要約
   - 使い方の例（コマンド実行例）
   - 今後の改善案や関連する次のステップ（あれば）
